```sql
CREATE TABLE users (
    user_id SERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    full_name VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    category VARCHAR(100),
    price DECIMAL(10,2) NOT NULL CHECK (price > 0),
```

### 2. Аналитические запросы

#### Запрос 1 – Заказы пользователей с итоговой суммой  

```sql
SELECT 
    o.order_id,
    u.full_name,
    o.order_date,
    o.status,
    SUM(oi.quantity * oi.unit_price) AS total_amount
FROM orders o
JOIN users u ON o.user_id = u.user_id
JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id, u.full_name, o.order_date, o.status
ORDER BY total_amount DESC;
```

#### Запрос 2 – Отчёт по категориям (выручка, количество)  

```sql
SELECT 
    p.category,
    SUM(oi.quantity) AS total_sold,
    SUM(oi.quantity * oi.unit_price) AS total_revenue
FROM order_items oi
JOIN products p ON oi.product_id = p.product_id
GROUP BY p.category
HAVING SUM(oi.quantity * oi.unit_price) > 10000;
```

#### Запрос 3 – Топ-3 пользователей с использованием CTE  

```sql
WITH user_totals AS (
    SELECT 
        u.user_id,
        u.full_name,
        SUM(oi.quantity * oi.unit_price) AS total_spent
    FROM users u
    JOIN orders o ON u.user_id = o.user_id
    JOIN order_items oi ON o.order_id = oi.order_id
    GROUP BY u.user_id, u.full_name
)
SELECT * FROM user_totals
ORDER BY total_spent DESC
LIMIT 3;
```

### 3. Оптимизация с индексом

**Индекс:**  
```sql
EXPLAIN ANALYZE
SELECT * FROM order_items WHERE order_id = 1;
```

**Сравнение:**  
- До индекса: *последовательное сканирование (Seq Scan)* – читается вся таблица.  
- После индекса: *индексное сканирование (Index Scan)* – читаются только нужные страницы.  
- Время выполнения сократилось с `X ms` до `Y ms`.

### 4. Ответы на вопросы (PostgreSQL)

**1. В чём разница между `WHERE` и `HAVING` в агрегатных запросах?**  
`WHERE` фильтрует строки **до** группировки (не может использовать агрегатные функции).  
`HAVING` фильтрует группы **после** группировки (может использовать `SUM`, `COUNT` и т.д.).  

**2. Зачем в запросе с `GROUP BY` нужно перечислять все неагрегированные столбцы?**  
Чтобы СУБД знала, как группировать строки. Если столбец указан в `SELECT`, но не в `GROUP BY` и не агрегирован, PostgreSQL не может однозначно определить, какое значение выбрать из группы – это приводит к ошибке или недетерминированному результату.

**3. Что означают `Seq Scan` и `Index Scan` в `EXPLAIN ANALYZE`? Как изменился план после добавления индекса?**  
- **Seq Scan** – последовательное чтение всей таблицы (медленно при больших объёмах).  
- **Index Scan** – чтение по индексу + обращение к таблице (быстро при селективной выборке).  
После создания индекса на `order_id` планировщик переключился с Seq Scan на Index Scan, что уменьшило количество прочитанных блоков и время выполнения.

### 2. CRUD-операции

#### 2.1 READ – заказы пользователя Alice с суммой заказа (используя `$lookup`)

```javascript
db.orders.aggregate([
    {
        $lookup: {
            from: "users",
            localField: "user_id",
            foreignField: "_id",
            as: "user_info"
        }
    },
    { $unwind: "$user_info" },
    { $match: { "user_info.email": "alice@example.com" } },
    {
        $addFields: {
            total_amount: {
                $reduce: {
                    input: "$items",
                    initialValue: 0,
                    in: { $add: ["$$value", { $multiply: ["$$this.quantity", "$$this.price"] }] }
                }
            }
        }
    },
    { $project: { order_id: "$_id", status: 1, total_amount: 1, items_count: { $size: "$items" } } }
]);
```

**Результат (ожидаемый):**

| order_id | status    | total_amount | items_count |
|----------|-----------|--------------|-------------|
| 1        | completed | 113000       | 3           |
| 3        | pending   | 1500         | 1           |

#### 2.2 UPDATE – добавить поле `discount` для заказов дороже 80000 руб.

```javascript
db.orders.aggregate([
    {
        $addFields: {
            total_amount: {
                $reduce: {
                    input: "$items",
                    initialValue: 0,
                    in: { $add: ["$$value", { $multiply: ["$$this.quantity", "$$this.price"] }] }
                }
            }
        }
    },
    { $match: { total_amount: { $gt: 80000 } } }
]).forEach(order => {
    db.orders.updateOne(
        { _id: order._id },
        { $set: { discount: 10, discount_reason: "large order" } }
    );
});
```

**Результат:** Заказ `_id: 1` (Alice, сумма 113000) получил `discount: 10`.

### 3. Агрегационный пайплайн

*Реализован пайплайн: `$unwind` → `$lookup` → `$group` → `$sort` → `$project`*

```javascript
db.orders.aggregate([
    { $unwind: "$items" },
    {
        $lookup: {
            from: "products",
            localField: "items.product_id",
            foreignField: "_id",
            as: "product"
        }
    },
    { $unwind: "$product" },
    {
        $group: {
            _id: "$product.category",
            total_sold: { $sum: "$items.quantity" },
            total_revenue: { $sum: { $multiply: ["$items.quantity", "$items.price"] } },
            avg_price: { $avg: "$items.price" }
        }
    },
    { $sort: { total_revenue: -1 } },
    {
        $project: {
            _id: 0,
            category: "$_id",
            total_sold: 1,
            total_revenue: 1,
            avg_price: { $round: ["$avg_price", 2] }
        }
    }
]);
```

**Результат (ожидаемый):**

| category     | total_sold | total_revenue | avg_price |
|--------------|------------|---------------|-----------|
| Электроника  | 5          | 116000        | 29166.67  |
| Книги        | 1          | 2500          | 2500.00   |

### 5. Ответы на вопросы (MongoDB vs PostgreSQL)

**1. Как в MongoDB представлена связь "заказ-товары" в отличие от PostgreSQL?**  
В PostgreSQL связь реализуется через отдельную таблицу `order_items` с внешними ключами.  
В MongoDB связь **документная** – массив `items` вложен прямо в документ `orders`. Это позволяет получать все данные заказа одним запросом без `JOIN` (но увеличивает дублирование).  

**2. В каком случае документная модель удобнее реляционной? Пример из работы.**  
Удобна, когда данные **иерархичны** и обращение к ним идёт "всё сразу". Например, получение заказа вместе со списком товаров – в MongoDB один запрос, в PostgreSQL – `JOIN` двух-трёх таблиц.  

**3. Какие операции в MongoDB оказались сложнее/проще по сравнению с SQL?**  
- **Проще:** вставка вложенных данных, отсутствие миграций (схема не фиксирована).  
- **Сложнее:** транзакции (многодокументные транзакции в MongoDB менее производительны, чем в PostgreSQL), сложные объединения (`$lookup` менее гибок, чем SQL `JOIN`).  

**4. Что такое `$unwind` и зачем он нужен?**  
`$unwind` "разворачивает" массив в отдельные документы. Необходим для группировки по элементам массива (например, каждый товар из заказа становится отдельным документом на время агрегации).

## Часть 3: Сравнение PostgreSQL и MongoDB (таблица заполняется по итогам работы)

| Характеристика | PostgreSQL | MongoDB |
|----------------|------------|---------|
| Схема | Фиксированная (DDL) | Гибкая (без схемы) |
| Связи | FOREIGN KEY, нормализация | Вложенные документы / ссылки |
| JOIN | `JOIN` между таблицами | `$lookup` или денормализация |
| Агрегация | `GROUP BY` + функции | `$group`, `$project`, `$unwind` |
| Транзакции | Полная ACID | ACID на уровне документа (многодокументные с версии 4.0) |
| Индексы | B-Tree, Hash, GIN, BRIN | Одиночные, составные, текстовые, гео |
| Скорость на связанных данных | Хорошо при нормализации | Лучше при чтении "одним куском" |
| Производительность записи | Требует обновления индексов | Быстрее для денормализованных структур |
